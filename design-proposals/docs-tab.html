Why this control exists

Every doc in context gets pulled into Coop's tool calls when relevant to the question. A 4 MB PDF deck eats ~8k tokens per query. Long or stale docs should be toggled off — the doc is still saved, Coop just won't reach for it.

Defaults: text and PDF are on. Images are off — Coop can't reliably read them without OCR. DOCX is on (Coop reads the extracted text even without an inline preview).

07

Open questions

4 calls for Matt
Q1Drag-drop: full-panel overlay during drag, or keep compact card?
Compact card reads calmer at rest but makes the drop target feel small. A full-panel overlay only appears while a drag is in flight — more forgiving, but adds a second UI state. Stance: start compact, add overlay only if users miss the target.
Q2Image docs: OCR to text so Coop can read?
Org charts, screenshots of email chains, even a photo of a handwritten note — OCR unlocks real value, but adds cost (API call per image) and complexity (provider choice, quality variance). Stance: ship without OCR; add behind a toggle later if users ask.
Q3Replace-doc flow: rename or overwrite?
If a user re-uploads Linear_JD.pdf after a description change, do we replace in place (preserving context-toggle state and position) or add as a new doc? Stance: "Replace" in the ⋮ menu swaps content, keeps metadata. Re-uploading from dropzone creates new (safer).
Q4DOCX: parse inline or always external open?
Libraries exist (mammoth.js) to render DOCX inline, but formatting is unreliable — tables break, styles drift, and it's ~300KB added. Coop can read the text regardless. Stance: external open, always. Revisit if users complain.
