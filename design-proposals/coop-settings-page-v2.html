How the dials actually change Coop

You asked whether these sliders are real or just for show. Here's the wiring: each slider divides the 0–1 range into five named anchors. Each anchor has a specific prompt-fragment. The three fragments get composed into a "Voice profile" block that prepends every system prompt. This is what actually lands on the API.

Composed system-prompt block

The three fragments get stitched into a Voice profile block that prepends every Coop call. This is the exact text the model sees. When you move a slider, only the matching fragment changes — not the wrapper.

Why this is "real," not cosmetic: the anchor text is the sole way Coop learns how you want the voice. The sliders don't set a style parameter on the API — they select which fragment goes into the system prompt. If the fragments are good, the voice changes. If the fragments are weak, the voice doesn't. Tuning is a prompt problem, not a UI problem.

Next-step idea

Surface-scoped voice profiles

Step 1 (this pass): one global voice profile. Same dials apply everywhere — chat, summaries, email drafts, scoring rationale.

Step 2 (future): let each surface inherit from global with override. Drafts probably want more warmth; summaries want more terse; scoring wants more direct. Same 3 dials per surface, but each surface lives under the Personality tab with its own dial panel, plus a "follow global" toggle.

Email drafts

Summaries

Chat · advice
Direct, moderately brief, casual — the sparring-partner register

Trigger for Step 2: ship Step 1, use it for 2+ weeks, confirm the global profile holds water. If you find yourself over- or under-shooting for one surface, that's the signal to split.

For you

Remaining open questions

1. Anchor count — 5 per dial, or 3?
v2 uses 5 anchors per slider (0–0.2 / 0.2–0.4 / 0.4–0.6 / 0.6–0.8 / 0.8–1.0). Five gives enough granularity to feel like the slider moves. Three would be simpler and harder to disappoint — but might feel like discrete buttons instead of a dial. My default is 5.
2. Show the composed prompt to the user, ever?
Option A: hide it entirely — users trust the slider. Option B: an "Advanced" disclosure under each slider showing the currently-selected fragment text (read-only). Option C: full prompt block shown here in the settings page, always-visible, so users know exactly what lands on the API. My default is B — disclosure on hover/click per slider — but C is the most honest.
3. Degraded-mode banner copy
When a user's only configured model rate-limits with no fallback, Coop shows a banner at top of chat. Current copy: "Haiku is over its rate limit. Add another model in Integrations to keep Coop responsive." Do we tell them which specific model would help most, or keep it generic?
4. Anchor text editability
Right now the 15 anchor fragments (3 dials × 5 positions) are hardcoded. Power-user move: let you edit them directly in Coop Settings → Personality → Advanced. Same risk as Operating Principles — power if done well, confusing if not. Hold for later?
