Tasks in pipeline overview — take 2

#239 · Second pass. The first crack produced three variations of "a strip of task chips somewhere on the page" — a band, a pill row, a right rail. None of them landed.

What I changed for take 2

Tasks aren't a strip — they're a list. I abandoned chip-grammar entirely. Reference products (Things 3's Today view, Linear's Inbox, Superhuman's triage) all share the same shape: vertical rows, typography carries it, inline-editable, minimal chrome. That's the right grammar for tasks because a task is fundamentally a line of text with a checkbox — not a card.

The real question

Not "where do tasks go on the overview" — "what's the top of the page for?" Today it mixes a data scoreboard (stat cards) with a to-do tray (chips). The reframe: the top is your day. Below it is the pipeline. Two jobs, two sections, clear hierarchy.

Three options, my pick is Option 1

Option 1 (recommended) gives tasks the top-of-page real estate. Option 2 splits the current overview in half (tasks left, stats right). Option 3 is the lightest touch — a collapsible drawer above the existing overview.

Recommended: Option 1 — tasks take the top, stats move below. Clean vertical list, inline-editable, no filter chips.

Tradeoff. Collapsible drawer is the safest move — no restructuring of the current section, tasks just get a dedicated home on top. Downside: drawers that collapse are things users ignore. If the default is collapsed, tasks become even more hidden than they are today. If the default is expanded, it's functionally Option 1 with an extra bar of chrome (the drawer header). So it's strictly worse than Option 1 in both states.

Why Option 1 wins

  • Hero placement that matches usage. When you open the pipeline overview in the morning, the question you're answering is "what do I need to do today?" — not "what's my application-count this week?". The top-of-page real estate goes to the job it's actually doing.
  • List grammar, not chip grammar. Tasks are one line of text with a checkbox. That's a row, not a card. Chips force title-truncation and a horizontal-scroll affordance; rows let you read the whole title at 14px without ornament.
  • Inline-editable by DESIGN.md convention. Matches the approved pattern (memory: feedback_inline_editable_pattern) — text reads-as-text, hover reveals actions, click text to edit in place.
  • Filter without chips. A quiet three-tab scope toggle (Today / This week / All) sits at the right of the section header — doesn't clutter the list. No "By company" filter — if you want company scoping, you go to the company detail page.
  • Monochrome first, color earned. Text is black/gray. The only colors: amber for today-due (brand accent, used sparingly), red for overdue (danger), green for done-check (success). Passes DESIGN.md "what does this color mean?" test.
  • Email-source handling without decoration. A 5px amber dot to the left of the check + inline "keep / dismiss" links. No orange left-border, no "from email" pill, no 3px decorative flag — the dot is enough.
  • Done-state truthful. Checkbox fills green on completion (200ms ease-out), row strikes through + fades to tertiary. "2 completed today · show" collapses done items; expanding restores them inline.
  • Empty state is a feature. Zero tasks due today renders "Nothing due today. Add a task →" — honest, no fake motivation, no gradient-confetti.

Rejected alternatives (don't re-raise)

  • Horizontal-scroll task chips — the previous pass's recommendation. Card/chip grammar, title-truncation, "scroll to see more" UX on a dashboard. Abandoned in take 2.
  • Filter chips (Today / This week / By company / Done) — the previous pass added four filter chips. Too much chrome; "By company" is useless on an overview; Done is just "show completed." Replaced with a single 3-tab scope toggle.
  • ⋯ menu as primary action target — previous pass had a ⋯ button on every card. Replaced with discoverable hover-reveal of snooze/open/more icons.
  • Right-rail variant — steals 280px of horizontal dashboard width; compresses stat row; off the table.
  • Separate /today page — tasks already have a /tasks page. Adding /today would duplicate. The overview's top is the summary; /tasks is the deep view.

Open questions for you

  • Do you want the "Today" header to carry the date prominently ("Today · Friday, Apr 17") or just the word ("Today")? Current mockup shows both.
  • When a task has a time ("2pm"), should it always show, or only when within 4 hours? Current mockup always shows.
  • Default scope: Today, or Today + overdue? Overdue-items-first is already the sort — scope "Today" includes overdue in the mockup.
