2 · Swipe momentum — video-game feel

Velocity-based commit, not distance-based

Today's swipe commits when you drag past ~30% of card width. v2 commits on velocity: quick flick + release → card flings off regardless of position. Slow drag still works too, matching the old behavior. Underpowered drags spring back with overshoot. The physics mean "quick throw, walk away" feels like a controller, not a mouse.

Three release modes. On release, we check (a) drag velocity in px/ms and (b) card position. One of three outcomes fires:
Why velocity matters: a fast, short flick is a stronger signal of "I've decided" than a slow drag to 30%. The slow drag says "I'm thinking"; the flick says "go." Treat them as different intents.
Mode 1
Flick commit
Quick drag + release above the velocity threshold. Card follows a curved arc offscreen with slight upward bias (throw feel). Next card springs in with mild overshoot.
FAST · COMMIT
trail fades over 180ms · card vanishes at screen edge
Mode 2
Slow drag commit
Deliberate drag past position threshold (~35% card width) and release. Card continues under its own momentum, linear-to-offscreen. The reliable, "I want to be sure" path.
SLOW · COMMIT
no arc · card slides to edge
Mode 3
Spring-back
Underpowered — slow release without reaching either threshold. Card springs back to center with overshoot (ease-spring 300ms). Tactile signal that your input didn't commit.
RETURN
small ghost = outward drift · card snaps back + overshoots slightly

Physics spec

Velocity threshold
0.9 px/ms horizontal — roughly "flicking" — tuned so a short 60px drag at 150ms+ commits.
Position threshold
35% of card width — reliable path when velocity is below threshold.
Commit arc
Flick mode uses a cubic Bézier with slight Y-lift — card arcs up ~30px then offscreen. Slow mode is linear. Both terminate at screen edge over --motion-lg (400ms).
Trail effect
Single semi-transparent ghost-card clones the departing card, decays opacity over 180ms — reads as motion blur without stacking.
Spring-back
Transform back to origin via --ease-spring (cubic-bezier(0.34, 1.56, 0.64, 1)), 300ms. The 1.56 coefficient is the overshoot — small tactile bounce.
Next card entry
After commit, next card animates from scale(0.96) translateY(8px) to rest with --ease-spring, 220ms. Subtle overshoot so the stack always feels alive.
Audio cue
Optional — tap into sounds.js for a soft whoosh on commit, muted when user mutes Coop sounds. Never on spring-back.
Reduced motion
Honor prefers-reduced-motion: reduce — skip the trail and the spring overshoot; use linear ease and half duration. Commit still works.

5 · Scoring strategy — Intention + Evidence + Through-line

A proposal for the scoring architecture you asked me to help with

You said you didn't want the math to be "red flags vs green flags, count-based" — if someone configures more reds than greens, scores shouldn't skew low. And Coop needs to find the through-line across experience, red flag, and green flag settings. Here's my proposal. Grounded in scoring.js's current structure — most of this is a prompt + post-processing change, not a new engine.

Three stages. Today's scoring runs one big LLM call that outputs dimension sub-scores with deltas from fired flags (scoring.js:391–475). v2 factors that into three stages — so each one is testable, and the math is clearer about what "through-line" means.
1
Intention synthesis
Coop reads your profile, preferences, red flags, green flags, compensation config, and pipeline history and produces a single short intention narrative — 3–5 sentences describing what you actually want in a role.

Example: "Senior or VP-level GTM leadership at Series B–D B2B SaaS. Revenue-focused scope (not pure product). $280K+ OTE floor; would take less at Seed–A stage with strong equity. Hard no on logistics SaaS."

Runs once per significant profile change. Cached on the entry as intentionNarrative.
›
2
Evidence collection
Per opportunity, per dimension: Coop enumerates evidence items — each one has {source, signal, valence, weight}.

Valence: positive / negative / neutral / N/A.
Weight: high / medium / low, set by the LLM based on (a) relevance to the intention narrative and (b) source credibility.

Flags configured by the user are a prompt input (their wording biases LLM attention) but not a direct math multiplier. Missing data → N/A, 0 weight. Never negative.
›
3
Through-line judgment
For each dimension, LLM reasons: "Given Matt's intention narrative, does this evidence support or undermine the dimension?"

Result: dimension score (1–10) + rationale + weighted evidence list.

Overall score = LLM-computed weighted composite, not a hard-coded dimension weighting. LLM can upweight / downweight dimensions based on the intention (e.g., if user's narrative emphasizes role scope, that dimension carries more weight for them specifically).

The red/green flag balance in config never mechanically pushes the score — the intention narrative is the source of truth for what matters.
// Pseudo-code for the scoring pipeline

function scoreOpportunity(entry) {
  // Stage 1 — synthesize intention once, cache on profile
  const intention = userProfile.intentionNarrative ??
    synthesizeIntention(profile, prefs, flags);

  // Stage 2 + 3 — one LLM call, structured output
  const result = await claudeApiCall({
    system: `You are scoring a job opportunity against the user's intention.
      For each dimension, collect evidence items with source, signal,
      valence, and relevance-weight. Return structured JSON.
      
      RULES:
      - Missing data = {valence:"na", weight:0}. Never negative.
      - Don't count flags. Judge how evidence aligns with intention.
      - Overall score weights dimensions per intention (not fixed).`,
    user: { intention, opportunity: entry, flags }
  });

  // Post-process — strip anything that doesn't cite evidence
  const cleaned = stripUngroundedClaims(result);

  return {
    score: cleaned.overall,
    intentionSnapshot: intention,
    dimensions: cleaned.dimensions, // each with evidence[], rationale, score
    rationale: cleaned.rationale
  };
}

6 · Current vs proposed — mapped to scoring.js

What actually changes in the code

The research read of scoring.js says evidence IS stored per signal today, but it's not first-class in the UI and the scoring math still treats missing signals inconsistently. This column pair makes the delta explicit so the Doer knows what to change.

Today

Current scoring

  • Dimensions: qualificationFit · roleFit · cultureFit · companyFit · compFit stored on entry.jobMatch.scoreBreakdown.
  • Flag math: fired flag contributes severity × 0.5 × (green ? +1 : -1) to its dimension (scoring.js:519–540).
  • Overall: fixed weighted average per scoringWeights (qual 20 · role 20 · culture 25 · company 20 · comp 15). Clamped [1,10].
  • Missing handling: strip flags with "no evidence" strings post-hoc (scoring.js:499–513). Good start but partial.
  • Evidence: stored as a string per fired flag / qualification. Not grouped by dimension; not surfaced in UI.
  • Intention: none. The full profile + prefs are passed as prompt context, but there's no synthesized "this is what Matt wants" narrative that the LLM explicitly reasons against.
  • Through-line: implicit in the LLM's reasoning but not structurally enforced. The prompt tells the model to score dimensions; it doesn't tell it to first find a coherent narrative about user intent.

Proposed

v2 scoring — Intention + Evidence + Through-line

  • Dimensions: same 5. Each dimension now stores {score, evidence[], rationale} instead of just a number + string.
  • Flag math: removed as direct multiplier. Flags become prompt context biasing LLM attention — the LLM decides if a fired flag is load-bearing for this opportunity.
  • Overall: LLM-weighted composite. Dimensions carry different weights per user intention. Fallback to fixed weights if LLM weighting confidence is low.
  • Missing handling: enforced universally. Any evidence item with valence:"na" has weight:0 and doesn't enter the math. UI shows N/A pill.
  • Evidence: first-class. Array per dimension, each with {source, signal, valence, weight, relevanceToIntention}. Surfaces in drill-down UI.
  • Intention: new. intentionNarrative synthesized from profile + prefs + flags, cached on profile, re-run on profile change. Pipeline history feeds updates.
  • Through-line: explicit in the system prompt. Every dimension rationale must reference how the evidence aligns with the intention narrative. No intention reference → reject in post-process.

7 · Open questions for you

1. Should the intention narrative be editable by the user?
Default: Coop synthesizes it from prefs + flags, shows it in Coop Settings, user can edit manually. Advantage: user gets direct control over what's load-bearing. Risk: stale edits override pref changes. My recommendation: show it read-only by default with a "refine" affordance that triggers re-synthesis; advanced users can force-edit the narrative text.
2. Do we keep the 5 dimensions or rethink them?
The 5 (Qualification / Role fit / Culture / Company / Comp) are durable, but v2's intention-driven weighting could naturally reveal dimension overlap or gaps. My default: keep the 5 for this pass, revisit if users report the weighting feels off.
3. How much cost does this add per score?
Stage 1 (intention) runs once per profile change — negligible. Stage 2 + 3 is still one LLM call per opportunity but with longer output (evidence arrays). Rough estimate: 20–40% more output tokens. Worth validating with a cost benchmark before rollout.
4. Rollout strategy?
New architecture is scored side-by-side with existing for ~10 opportunities, diff checked manually, then cutover. Legacy scoreBreakdown / flagsFired fields stay for 1–2 weeks as fallback read, then removed. Matches how v1 scoring overhaul rolled out per prds/shipped/S4-scoring-overhaul.md.
5. Swipe physics — tune the velocity threshold together?
0.9 px/ms is a reasonable starting point but feel-testing will tell us if it's too twitchy or too sleepy. Suggest a dev-mode overlay that shows live velocity readings during the first week so we can calibrate without guessing.
