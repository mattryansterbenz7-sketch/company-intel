View switcher becomes Kanban / Table only.
One clearer primary mental model, one dense audit surface.
Why: Kanban covers "where is everything" visually. Table covers "let me sort, search, scan everything at once." Grid sits between them with less info than Kanban (no next step / action context) and less density than Table (fewer rows per page). No moment where it's the right tool.
7Open questions

Three calls for Matt before spec

  1. Tasks drawer default state. Collapsed on first load (shown in hero) — preserves main-view real estate, user expands when they want it. Alt: expanded by default so users discover it. Gut: collapsed. Remember last-used state per session.
  2. Stat-card filter persistence. Persistent across view switches (click "Interviewing" in Kanban → switch to Table → still filtered to Interviewing) or reset on switch? Gut: persistent. Filters are state about the data, views are state about the visualization — they're orthogonal.
  3. Grid view. Kill (Kanban + Table only) or keep (polish it)? Gut: kill. No moment where it beats Kanban or Table. Switcher simplifies to a 2-state toggle. If kept, cards get a DESIGN.md pass but we own a third surface forever.

Reply "ship it" if the three gut-calls look right, otherwise name which to change.
