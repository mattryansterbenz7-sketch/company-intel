Stage label treatment — #242 V2

Stage labels appear across Kanban headers, Inbox chips, Status filters, the Company detail picker, Activity feed, Apply Queue, and Coop chat. Today they're too long and inconsistent — columns wrap, labels get cut off at the right edge, and a stray checkmark on "Applied" breaks the grammar. This is the recommended fix: a tighter taxonomy, a primary + substate pattern for the dual-state stages, and a single consistent visual treatment on every surface.

Matt: redesign stage name display across all surfaces. Rename where too long, separate substates, apply the same pattern everywhere.

What changed in v2

  • Q1 resolved: semantic tint on substate pills is approved. Scheduled = soft blue, Held = soft teal. Kept.
  • Q2 resolved, NEW SECTION 5.5 added: "Why lost" is not a column and not a substate. It's an attribute/tag on the opportunity itself. Closed Lost is one primary stage; an optional lostReason field on the entry carries the "why." Renders as a chip on the card / filter in Kanban.
  • Q3 remains: Apply Queue density / icons — still flagged as a follow-up if density becomes a real problem.
Recommended approach
1. Recommended taxonomy

Rename aggressively for brevity. The rule: short, concrete, and parseable at a glance. Split Scheduled/Held out as substates rather than independent stages.

5. Data migration note

For the Doer handoff — how to apply this without breaking existing entries.

Keep stageKey stable. Display labels live in the pipeline config, not on the entries. Rename only the displayed string — reached_out_requested_intro still exists as an internal key, it just renders as "Outreach".

Introduce stageSubstate on entries. New optional string field, one of "scheduled" | "held" | null. Only relevant for stages that declare a substate in their pipeline config (supportsSubstate: true).

Migration for split stages. Entries currently in initial_call_scheduled → set stageKey: "initial_call", stageSubstate: "scheduled". Same pattern for initial_call_held, hm_call_scheduled, hm_call_held. Gate behind a one-shot migration flag (_migratedStageSubstateV1) in the startup chain alongside existing guards.

Stage timestamps. Generic stageTimestamps keeps working — the map stays keyed by the new consolidated stageKey. Pre-migration timestamps for split-stage keys get merged: keep the earliest entered timestamp under the new primary key, drop the scheduled/held-specific keys (the substate is transient, not a timestamped event).

Pipeline config schema change. Each stage entry in the pipeline config gains three optional fields: dotColor, supportsSubstate, substateOptions: [{ key, label }]. Default pipeline ships with the new schema; custom user pipelines get migrated in-place with sensible defaults (no substate).

5.5. "Why lost" — as an opportunity attribute, not a column

Matt's direction: "why it was lost is not a column, it should be an attribute / property or tag within the opportunity itself." The reason is data ON the entry, not a stage slot. Saves Kanban from blooming into 4 terminal columns; keeps the semantic weight intact where it belongs — the opportunity record.

Where lostReason renders
  • Kanban card (shown left) — colored pill below company/role when the entry is in Closed Lost
  • Opportunity detail header — next to the stage pill when stage is Closed Lost
  • Inbox row — small chip trailing the stage badge
  • Kanban column header filter bar (shown right) — filter chips appear above the Closed Lost column when it has ≥2 entries, for quick "show me the ghosts" / "the rejects"
  • Coop context — when Coop references a Closed Lost entry, it can cite the reason ("Voxel rejected you") instead of just "Voxel is Closed Lost"
  • Stats / dashboard (future) — breakdown of why opportunities died is a durable analytics signal
Migration: lostReason is a new optional string field on the entry. Existing entries in Closed Lost get lostReason: null. User can set it from the opportunity detail page or directly on the Kanban card via a small "+ why?" affordance on the card footer when reason is unset.

Apply Queue density — revisit stage icons?

This proposal rejects icons everywhere.
